# Advisor Orchestrator Handoff

## 目的

Gemini CLI / Codex CLI / Claude Code を「相談役」として使い、
1つのテーマについてそれぞれの意見を収集し、比較し、統合し、
必要に応じて実装タスクを分解して worker に割り当て、
最終的にレポートを返す仕組みを新規フォルダで作りたい。

このプロジェクトは、一般的な multi-agent 開発環境をそのまま再現するのではなく、
「3つのモデルを相談役として使うための軽量オーケストレーション」を目指す。

## 背景

- 参考リポジトリ: `yohey-w/multi-agent-shogun`
- 参考にしたい点:
  - 役割分離
  - オーケストレーションを外部状態で管理する発想
  - コンテキスト階層の分離
  - CLI 非依存の設計
- そのままは不要な点:
  - 多人数前提の深い階層
  - 大きな `tmux` 運用
  - Android 連携
  - 重い常時監視ダッシュボード

## ユーザー前提

- ユーザーは個人開発者に近い使い方を想定
- 本業は高校の理科教員
- Gemini / Codex / Claude Code をすでに使い倒している
- 各 CLI は 20 ドル帯の契約で利用中
- 「実装人数が必要」というより、
  まずは複数モデルの意見をまとめる相談システムを作りたい

## 作りたいもの

### コア要件

1. 1つの課題や要望を入力する
2. Codex / Claude / Gemini に同じ論点を投げる
3. それぞれの意見を構造化して保存する
4. 3者の一致点・相違点・保留点を抽出する
5. 必要なら差分だけ再質問する
6. 統合案を作る
7. 実装が必要ならタスク分割して worker に振る
8. 最終レポートを返す

### 重要な方針

- 3者をいきなり自由会話させない
- まずは各モデルに独立で回答させる
- 回答フォーマットを揃える
- 無理に全会一致にしない
- 少数意見も残す
- まずは相談役機能を MVP にする
- worker 自動分担は後から足す

## 推奨アーキテクチャ

### 役割

- `orchestrator`
  - 全体進行
  - 入力正規化
  - 各役割の呼び出し
- `advisors`
  - `codex`
  - `claude`
  - `gemini`
- `synthesizer`
  - 3者の意見比較
  - 合意案、対立点、未解決点の整理
- `planner`
  - 実装が必要な場合のタスク分割
  - 必要人数の見積もり
- `workers`
  - 実装担当
  - 必要時のみ起動
- `reviewer`
  - 実装横断レビュー
  - 最終報告の整形

### フロー

1. ユーザー入力を `problem statement` に正規化
2. 3 CLI に同一テンプレートで相談
3. 回答保存
4. 差分抽出
5. 差分のみ再質問
6. 統合方針の決定
7. 必要なら実装計画生成
8. 必要なら worker 実行
9. 最終レポート出力

## 相談役の出力フォーマット案

```yaml
summary: ""
approach:
  - ""
tradeoffs:
  - pro: ""
    con: ""
risks:
  - ""
unknowns:
  - ""
recommended_next_steps:
  - ""
confidence: 0-100
```

この形式により、各モデルの比較対象を揃えやすくする。

## MVP の範囲

最初に作るべきものは、実装班つきのフルシステムではなく、
「相談役システム」の最小版。

### MVP に含めるもの

- 3つの CLI を同一インターフェースで呼ぶ adapter
- 同じ質問テンプレートで意見を取る仕組み
- 回答をファイル保存する仕組み
- 差分比較
- 統合レポート生成

### MVP に含めないもの

- 大規模な並列 worker 管理
- 常時監視ダッシュボード
- Android 連携
- 複雑な永続メモリ

## 推奨ディレクトリ構成

```text
advisor-orchestrator/

        request.yaml
        advisor_codex.yaml
        advisor_claude.yaml
        advisor_gemini.yaml

        plan.yaml
        report.md
```

## 実装順の提案

1. `providers/` で Codex / Claude / Gemini を同じインターフェースで呼べるようにする
2. `prompts/advisor.md` を決める
3. 単一の依頼を 3者に配って YAML 回答を得る
4. 差分比較ロジックを作る
5. 統合レポートを作る
6. そのあとで planner / workers を追加する

## 実装上の考え方

- 3者に自由討論させるより、
  司会が順番に意見を集めて再質問する方式のほうが収束しやすい
- 「討論システム」ではなく、
  「構造化レビュー会議システム」として設計した方が扱いやすい
- orchestration はアプリ内部に埋め込みすぎず、
  ファイルや YAML で状態を外出しした方がデバッグしやすい
- CLI の違いは provider 層に閉じ込める

## 会話コンテキストに関するメモ

- 同じチャットスレッド内なら、作業フォルダが変わっても会話の文脈は引き継げる
- ただし、フォルダ固有のファイルやローカル指示は別途読み直しが必要
- 別スレッドや別セッションでは基本的に引き継がれない前提で考える

## 次の作業でやると良いこと

新フォルダに移ったら、まず以下をやる。

1. 新規プロジェクトフォルダを作る
2. 最低限の言語と実行方法を決める
3. `providers/` のインターフェースを先に固定する
4. `advisor` 用プロンプトテンプレートを作る
5. 1回の相談をファイル保存できるところまで作る

## 補足

このプロジェクトの本質は、
「複数モデルで実装を分担すること」そのものより、
「複数モデルの意見を比較し、意思決定の質を上げること」にある。

最初の価値は相談役機能にあるため、
MVP では worker 自動生成よりも、
意見比較と統合の品質を優先する。
